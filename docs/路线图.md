## 现状（2026-05）

服务端 reactor 主干已完成：

```
include/base/   noncopyable  timestamp  logger  buffer  currentthread  thread
include/net/    callbacks  inetaddress  socket  channel  poller  epollpoller
                eventloop  acceptor  tcpconnection  tcpserver
                eventloopthread  eventloopthreadpool
```

benchmark 在 loopback 上能跑出 ~2GB/s 吞吐。

---

## 与 muduo 对齐还差什么

| 类别 | 缺失内容 | 影响 |
|------|----------|------|
| 定时能力 | `EventLoop` 不支持 `runAt/runAfter/runEvery/cancel` | 无法做心跳、超时、退避重连、空闲淘汰 |
| 客户端能力 | 没有 `Connector` / `TcpClient` | 只能做被动服务端，无法做主动连接 |
| 生产化细节 | 未忽略 `SIGPIPE`、`Acceptor` 未处理 `EMFILE` | 偶发崩溃 / fd 耗尽时 CPU 飙满 |
| 派生能力 | 空闲连接淘汰、断线重连示例 | 缺少真实场景演示 |

```
阶段 1: Timer / TimerQueue        ← 基础设施，最先做
   │
```

## 阶段 1：Timer / TimerQueue

### 目标
`EventLoop` 暴露四个公共接口：
```cpp
TimerId runAt(Timestamp when, TimerCallback cb);
TimerId runAfter(double delaySec, TimerCallback cb);
TimerId runEvery(double intervalSec, TimerCallback cb);
void    cancel(TimerId timerId);
```

### 学习要点
- **`timerfd_create` + `timerfd_settime`**：核心 Linux idiom。把"等待到某时刻"转成"等待 fd 可读"，复用现有 `Channel` + `Poller`
- **`std::set<std::pair<Timestamp, Timer*>>` 有序队列**：插入/取最早 O(log N)，相对小顶堆更易实现 cancel
- **TimerId 的延迟取消**：保存 `Timer*` + sequence；如果该 Timer 已在"正在到期处理"列表中，标记为 cancellingTimers_，下一轮跳过重新插入
- **周期定时**：到期回调执行完后，若 repeat=true 更新到期时间并重新插入

### 涉及文件
- 新增 `include/net/timer.hpp`、`include/net/timerid.hpp`、`include/net/timerqueue.hpp`
- 新增 `src/net/timer.cpp`、`src/net/timerqueue.cpp`
- 修改 `include/net/eventloop.hpp` / `src/net/eventloop.cpp`：持有 `std::unique_ptr<TimerQueue>`，新增 4 个 public 接口
- 修改 `CMakeLists.txt`：添加两个新源文件
- （可选）`include/net/callbacks.hpp`：添加 `using TimerCallback = std::function<void()>;`

### 复用的现有设施
- `Channel`（包装 timerfd 注册到 Poller）
- `Timestamp::now()`（已有微秒精度）
- `EventLoop::runInLoop`（跨线程添加 timer 必须经过 loop 线程）

### 验证
- 新增 `benchmark/timer_test.cpp`：
  - `runAfter(1s, ...)` 触发顺序断言
  - `runEvery(100ms, ...)` 计 N 次后 cancel
  - 跨线程 `runAfter` 投递
- `src/main.cpp` 加 `loop.runEvery(5.0, []{ LOGINFO("alive"); })`

> 详细设计见 `阶段1-timer设计.md`

## 阶段 2：SIGPIPE 忽略 + 信号工具

### 目标
- **最低**：进程启动一次性 `signal(SIGPIPE, SIG_IGN)`，避免对端 RST 后 `send` 把进程打死
- **可选**：`signalfd` 把 SIGINT/SIGTERM 转为 `EventLoop` 任务，实现优雅退出

### 学习要点
- `SIGPIPE` 产生时机：写已被对方关闭的 fd；忽略后 `send` 返回 `-1`、`errno=EPIPE`
- 全局静态对象在 `main` 之前完成构造（无侵入安装信号忽略）
- `signalfd` 把信号化为 fd 可读，融入 reactor

### 涉及文件
- 最小版：新增 `include/base/sigpipe_ignorer.hpp`，匿名命名空间的静态对象在构造函数里调用 `signal(SIGPIPE, SIG_IGN)`；由 `logger.hpp` 或 `tcpserver.hpp` include 它
- 完整版（可选）：新增 `include/net/signaler.hpp` + .cpp

### 验证
- 客户端 `connect` 后立刻 `close`，服务端立刻 `send` 大块数据
- 预期 server 进程不退出，日志显示 EPIPE

## 阶段 3：Connector + TcpClient

### 目标
对称 `TcpServer`：
```cpp
EventLoop loop;
TcpClient client(&loop, addr, "echo-client");
client.setMessageCallback(...);
client.connect();
loop.loop();
```

### 学习要点
- **非阻塞 connect**：立即返回 `-1`、`errno=EINPROGRESS`；之后必须监听写事件，写事件到来时 `getsockopt(fd, SOL_SOCKET, SO_ERROR, ...)` 取真实结果
- **errno 分类处理**：
  - `EINPROGRESS / EINTR / EISCONN` → 成功
  - `EAGAIN / EADDRINUSE / EADDRNOTAVAIL / ECONNREFUSED / ENETUNREACH` → retry
  - 其它 → 放弃
- **指数退避**：初始 500ms，每次失败翻倍，封顶 30s；依赖阶段 1 的 `runAfter`
- **状态机**：`kDisconnected → kConnecting → kConnected`；重连前必须 `close` 旧 sockfd 重建
- **复用关键点**：连接建立后**直接复用现有 TcpConnection**，无需改造

### 涉及文件
- 新增 `include/net/connector.hpp`、`include/net/tcpclient.hpp`
- 新增 `src/net/connector.cpp`、`src/net/tcpclient.cpp`
- 修改 `CMakeLists.txt`：添加两个新源文件
- （可能）修改 `include/net/socket.hpp`：加一个 `connect(const InetAddress&)` 包装

### 复用的现有设施
- `TcpConnection`（连接建立后照搬，无需改造）
- `Channel` / `EventLoop` / `Socket`
- 阶段 1 的 `EventLoop::runAfter`（用于退避）

### 验证
- 新增 `benchmark/echo_client.cpp`：用 `TcpClient` 连 8888 端口的 echo-server，多线程并发收发
- 重连验证：手动 `kill` 服务端，看客户端日志显示退避延迟逐步增大；重启服务端，客户端自动恢复

## 阶段 4：Acceptor EMFILE 防护

### 目标
fd 耗尽 (`accept` 返回 `EMFILE`) 时不让 epoll 陷入可读事件死循环烧 CPU。

### 学习要点
- 套路：进程启动预占一个 idle fd（`open("/dev/null", O_RDONLY)`）；`accept` 失败且 `errno==EMFILE` 时：**close idle → accept → close 新连接 → 重新 open idle**
- 边界：仅缓解，不能"治本"；真正修复要调高 ulimit 或加限流

### 涉及文件
- 修改 `include/net/acceptor.hpp`：新增 `int idleFd_`
- 修改 `src/net/acceptor.cpp`：构造时 `open("/dev/null", ...)`，`handleRead` 增加 `EMFILE` 分支

### 验证
- `ulimit -n 32` 启动 server，并发 50 个连接进来，观察日志和 CPU 占用（不应飙满）

## 阶段 5：空闲连接淘汰（演示性，可选）

### 目标
demo 层演示"N 秒无数据自动关闭"，**不侵入 TcpServer 内核**，体现 muduo "复用 Timer + weak_ptr" 的设计哲学。

### 学习要点
- 时间轮：固定 N 个 `std::unordered_set<weak_ptr<TcpConnection>>` 循环桶，每秒 `runEvery` 推进一格；最老一格自动析构里面所有 weak 引用——若连接在期间没"刷新"，shared_ptr 在桶里就是它唯一引用，自动断开
- 连接收到消息时把自己重新放进当前桶
- 用 weak_ptr 而非 shared_ptr 防止干扰连接正常生命周期

### 涉及文件
- 新增 `examples/idle_server.cpp`（或 `benchmark/idle_server.cpp`），不改核心库
- 依赖阶段 1 的 `EventLoop::runEvery`

### 验证
- 启动 demo，客户端 `nc 127.0.0.1 9999` 后什么也不发，N 秒后被服务端踢掉

## 推进规范

- **每阶段一个独立提交或分支**：完成 → 自测 → 提交 → 更新笔记 → 开下一阶段
- **每阶段后更新 `docs/个人笔记.md`**：增加对应章节（如"timerfd 与 epoll 融合"、"非阻塞 connect 三态判定"）

- **不要跳过阶段 1**：阶段 3 / 5 都强依赖 Timer
- **暂不引入**：HTTP / Codec / 异步日志 / 单元测试框架——属于"扩展层"而非 reactor 核心，留给后续路线图
