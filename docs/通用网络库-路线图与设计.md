# tinynet · 通用网络库 路线图与设计文档

> 目标定位：**Linux 生产级全功能**（epoll 专注，不做跨平台/协程）。
> 本文给出路线图、各组件的设计与 **API 声明**；**具体实现（.cpp）由你完成**。
> 所有 API 均贴合现有风格：`noncopyable` 基类、`xxx_` 成员、`kXxx` 常量、`runInLoop` 跨线程投递、回调走 `callbacks.hpp` 的 `using` 别名。
> **命名空间**：本库所有类型都位于 `namespace tinynet` 内（下文 API 代码块为简洁省略了 `namespace tinynet { ... }` 包裹，落地时记得补上）。

---

## 0. 设计哲学（贯穿全文的一把尺子）

> **协议无关、任何上层都用得到的 → 下沉到 net 库；和某个协议绑定的 → 上浮到应用层。**

三层结构始终保持清晰：

```
应用层        业务逻辑、聊天/文件协议、断点续传        ← 你的 demo / 测试
   ↑
编解码层      LengthHeaderCodec(分帧)、(可选)protobuf   ← 独立一层，不进 TcpConnection
   ↑
传输层(net)   TcpServer/Client · UdpServer · TimerQueue ← 本库的边界，只管字节流/数据报
   ↑
Reactor 核心  EventLoop · Poller · Channel · 线程池     ← 协议无关，已完成，原样复用
```

线程模型不变：**one loop per thread**。所有"可跨线程调用"的接口，内部都用 `runInLoop / queueInLoop` 把真正的操作投递回对象所属的 loop 线程执行。

## 1. 路线图总览

| 阶段 | 组件 | 性质 | 依赖 |
|---|---|---|---|
| **1a** | `TimerQueue` + `EventLoop::runAt/After/Every/cancel` | 关键路径起点 | — |
| **1b** | `Connector`（非阻塞 connect + 指数退避重连） | 客户端基石 | 1a |
| **1c** | `TcpClient`（对称于 TcpServer） | 双向通用里程碑 | 1b |
| **2a** | `UdpServer`（数据报，平行于 TCP） | 传输扩展 | — |
| **2b** | Unix domain socket（复用 TCP 路径） | 传输扩展 | — |
| **2c** | `send` 重载 + zero-copy `sendFile` | 标配缺口 | — |
| **3a** | `LengthHeaderCodec` + `Buffer` 整数读写 | 上层分帧 | — |
| **3b** | `signalfd` 优雅退出 | 运维 | — |
| **3c** | 心跳 / 空闲连接踢除（TimingWheel） | 运维 | 1a |

**主干**：`1a → 1b → 1c`。做完这条线，库就从"只能写服务端"变成 **client + server 双向通用**，这是第一个真正的里程碑。

## 2. 阶段一 · 双向通用里程碑

### 2.1 组件 1a：定时器 `TimerQueue`

#### 职责
把"未来某个时刻执行回调"的能力交给 EventLoop。核心技巧：用 **`timerfd`** 把"时间到达"变成一个 **fd 可读事件**，从而天然融入你现有的 Reactor —— 不需要修改 `Poller::poll` 的超时参数，定时器就是又一个挂在 Poller 上的 Channel。

#### 前置改动：`Timestamp` 需要补运算能力
现在的 `Timestamp` 只有 `now()` 和 `toString()`，无法比较、无法做加法，进不了有序容器。需要新增：

```cpp
// include/base/timestamp.hpp 内补充
class Timestamp {
 public:
  // ... 现有内容 ...

  /** @brief 返回内部微秒数，供比较与运算使用。 */
  int64_t microSecondsSinceEpoch() const { return microSecondsSinceEpoch_; }

  /** @brief 时间戳是否有效（>0）。 */
  bool valid() const { return microSecondsSinceEpoch_ > 0; }

  /** @brief 返回一个无效时间戳（值为 0）。 */
  static Timestamp invalid() { return Timestamp(); }
};

/** @brief 比较两个时间戳的先后。 */
inline bool operator<(Timestamp lhs, Timestamp rhs) {
  return lhs.microSecondsSinceEpoch() < rhs.microSecondsSinceEpoch();
}
inline bool operator==(Timestamp lhs, Timestamp rhs) {
  return lhs.microSecondsSinceEpoch() == rhs.microSecondsSinceEpoch();
}

/**
 * @brief 在时间戳上增加若干秒（可为小数）。
 * @param timestamp 基准时间。
 * @param seconds   增加的秒数。
 * @return 新的时间戳。
 */
inline Timestamp addTime(Timestamp timestamp, double seconds) {
  int64_t delta = static_cast<int64_t>(seconds * Timestamp::kMicroSecondsPerSecond);
  return Timestamp(timestamp.microSecondsSinceEpoch() + delta);
}
```
> 注意：`kMicroSecondsPerSecond` 目前是 private，`addTime` 要用就得把它放到 public，或在 `.cpp` 里实现 `addTime`。

#### 回调类型（加进 `callbacks.hpp`）
```cpp
/** @brief 定时器到期时执行的回调。 */
using TimerCallback = std::function<void()>;
```

#### `Timer`（内部类，不对用户暴露）
表示一个定时任务本身。

```cpp
// include/net/timer.hpp
class Timer : noncopyable {
 public:
  /**
   * @param cb       到期回调。
   * @param when     首次到期时刻。
   * @param interval 重复间隔（秒）；<=0 表示一次性定时器。
   */
  Timer(TimerCallback cb, Timestamp when, double interval);

  /** @brief 执行回调。 */
  void run() const { callback_(); }

  Timestamp expiration() const { return expiration_; }
  bool repeat() const { return repeat_; }
  int64_t sequence() const { return sequence_; }

  /** @brief 周期定时器：以 now 为基准重新计算下次到期时刻。 */
  void restart(Timestamp now);

 private:
  const TimerCallback callback_;
  Timestamp expiration_;          // 注意：restart 会改它，所以不是 const
  const double interval_;
  const bool repeat_;
  const int64_t sequence_;        // 全局唯一自增序号

  static std::atomic<int64_t> s_numCreated_;  // 生成 sequence_
};
```

#### `TimerId`（对外句柄，用于 `cancel`）
```cpp
// include/net/timerid.hpp
class TimerId {
 public:
  TimerId() : timer_(nullptr), sequence_(0) {}
  TimerId(Timer* timer, int64_t seq) : timer_(timer), sequence_(seq) {}

  friend class TimerQueue;   // 只有 TimerQueue 能读私有成员

 private:
  Timer* timer_;
  int64_t sequence_;
};
```
> **为什么 TimerId 要带 `sequence_` 而不只是裸指针**：一次性定时器到期后会被 `delete`，其地址可能被新 Timer 复用。若 `cancel` 只比指针，会误删新定时器（ABA 问题）。`(Timer*, sequence)` 二元组才能唯一标识一个定时器。

#### `TimerQueue`
```cpp
// include/net/timerqueue.hpp
class TimerQueue : noncopyable {
 public:
  explicit TimerQueue(EventLoop* loop);
  ~TimerQueue();

  /**
   * @brief 注册一个定时器（线程安全，可跨线程调用）。
   * @return TimerId，用于后续 cancel。
   */
  TimerId addTimer(TimerCallback cb, Timestamp when, double interval);

  /** @brief 取消一个定时器（线程安全）。 */
  void cancel(TimerId timerId);

 private:
  using Entry = std::pair<Timestamp, Timer*>;        // 按到期时间排序
  using TimerList = std::set<Entry>;
  using ActiveTimer = std::pair<Timer*, int64_t>;    // 按指针排序，供 cancel
  using ActiveTimerSet = std::set<ActiveTimer>;

  // 以下都在 loop 线程执行
  void addTimerInLoop(Timer* timer);
  void cancelInLoop(TimerId timerId);
  void handleRead();                                 // timerfd 可读回调

  std::vector<Entry> getExpired(Timestamp now);      // 取出所有已到期
  void reset(const std::vector<Entry>& expired, Timestamp now);  // 周期性的重新入队
  bool insert(Timer* timer);                         // 返回：是否改变了最早到期时刻

  EventLoop* loop_;
  const int timerfd_;
  Channel timerfdChannel_;

  TimerList timers_;                                 // 主索引：按时间
  ActiveTimerSet activeTimers_;                      // 辅索引：按指针（同一批 Timer*）

  // —— 处理"在定时器回调里 cancel 定时器"的自注销难题 ——
  std::atomic<bool> callingExpiredTimers_;
  ActiveTimerSet cancelingTimers_;
};
```

#### 实现要点 / 易错点
1. **timerfd 三件套**：`timerfd_create(CLOCK_MONOTONIC, TFD_NONBLOCK | TFD_CLOEXEC)` 建 fd；`timerfd_settime` 设置下次触发的绝对/相对时间；到期后 fd 可读，`handleRead` 里 `read()` 8 字节清除事件（不读会一直触发）。
2. **两个 set 同步维护**：`timers_` 按 `(到期时间, 指针)` 排序，用来取最早；`activeTimers_` 按 `(指针, sequence)` 排序，用来 O(log n) 定位待 cancel 的 Timer。两者元素一一对应，增删要同步。
3. **`handleRead` 主流程**：读 timerfd → `getExpired(now)` 取出所有 `expiration <= now` 的 → 逐个 `run()` → `reset()` 把其中 `repeat()` 的 `restart` 后重新 `insert`，一次性的 `delete` → 用 `timers_` 的新队首重新 `timerfd_settime`。
4. ⚠️ **自注销难题（重点）**：回调里可能 `cancel` 掉"正在执行的定时器"或"同批里还没执行的定时器"。用 `callingExpiredTimers_` 标记"正在跑回调"，`cancelInLoop` 发现要取消的 timer 正在这批里就塞进 `cancelingTimers_`，`reset` 时跳过它们、不重新入队。**这是整个 TimerQueue 最容易写错的地方，建议单独写测试。**
5. **线程安全**：`addTimer/cancel` 可跨线程，内部一律 `runInLoop(addTimerInLoop / cancelInLoop)`，因此 `timers_` 等成员**无需加锁**——它们只被 loop 线程访问。这是 Reactor 模型"用线程归属代替锁"的典型范式，也是本库一以贯之的线程安全策略。

#### `EventLoop` 集成
```cpp
// include/net/eventloop.hpp 内补充
class EventLoop {
 public:
  // ...
  /** @brief 在指定时刻执行一次回调。 */
  TimerId runAt(Timestamp time, TimerCallback cb);
  /** @brief delay 秒后执行一次回调。 */
  TimerId runAfter(double delay, TimerCallback cb);
  /** @brief 每隔 interval 秒重复执行回调。 */
  TimerId runEvery(double interval, TimerCallback cb);
  /** @brief 取消定时器。 */
  void cancel(TimerId timerId);

 private:
  std::unique_ptr<TimerQueue> timerQueue_;   // 在构造函数里 new
};
```

#### ✅ 验收标准
- `runEvery(1.0, ...)` 每秒打印一次；`runAfter(3.0, ...)` 只触发一次；`cancel` 能成功阻止未到期定时器；在回调里 cancel 自己不崩溃。

### 2.2 组件 1b：`Connector`

#### 职责
客户端侧的"反 Acceptor"：主动发起一条 TCP 连接。难点全在**非阻塞 connect 的状态机**和**断线退避重连**上。

#### ⚠️ 非阻塞 connect 的坑（重点）
- 非阻塞 `connect()` 通常立即返回 `-1` 且 `errno == EINPROGRESS`，连接是否成功要**异步**得知。
- 连接结果通过 socket 的 **可写事件** 通知（不是可读！）。
- **可写 ≠ 成功**：必须 `getsockopt(fd, SOL_SOCKET, SO_ERROR, ...)` 检查，err==0 才是真成功。
- **自连接 self-connect**：客户端可能连上自己（源端口恰好等于目标端口），需检查 `localAddr == peerAddr`，命中则重连。
- **退避重连**：失败后延时重连，延时指数增长 `0.5s → 1s → 2s → … → 30s` 封顶。**这就是 Connector 依赖 TimerQueue（`runAfter`）的原因。**
- errno 分类：`EINPROGRESS/EINTR/EISCONN` 视为进行中/已连接；`EAGAIN/EADDRINUSE/ECONNREFUSED/ENETUNREACH...` 视为失败需重连；其余记录错误。

#### API
```cpp
// include/net/connector.hpp
class Connector : noncopyable, public std::enable_shared_from_this<Connector> {
 public:
  /** @brief 连接建立成功后，把已连接 sockfd 交给上层（TcpClient）。 */
  using NewConnectionCallback = std::function<void(int sockfd)>;

  Connector(EventLoop* loop, const InetAddress& serverAddr);
  ~Connector();

  void setNewConnectionCallback(const NewConnectionCallback& cb) {
    newConnectionCallback_ = cb;
  }

  void start();    ///< 发起连接，可任意线程调用
  void restart();  ///< 重连，必须在 loop 线程
  void stop();     ///< 停止连接/重连，可任意线程调用

  const InetAddress& serverAddress() const { return serverAddr_; }

 private:
  enum States { kDisconnected, kConnecting, kConnected };
  static constexpr int kMaxRetryDelayMs  = 30 * 1000;
  static constexpr int kInitRetryDelayMs = 500;

  void setState(States s) { state_ = s; }
  void startInLoop();
  void stopInLoop();
  void connect();               ///< 创建非阻塞 socket 并 ::connect
  void connecting(int sockfd);  ///< 进入 kConnecting，注册可写事件
  void handleWrite();           ///< 可写：查 SO_ERROR，成功→交付，失败→retry
  void handleError();
  void retry(int sockfd);       ///< 关闭 fd，runAfter 后重连，延时翻倍
  int removeAndResetChannel();  ///< 从 Poller 摘除 channel_ 并返回 fd
  void resetChannel();

  EventLoop* loop_;
  InetAddress serverAddr_;
  bool connect_;                ///< stop() 后置 false，阻止重连
  States state_;
  std::unique_ptr<Channel> channel_;   ///< 每次重连重建（fd 变了）
  NewConnectionCallback newConnectionCallback_;
  int retryDelayMs_;
};

using ConnectorPtr = std::shared_ptr<Connector>;   // 加进 callbacks.hpp
```

#### 实现要点
- **为何 `enable_shared_from_this`**：`retry` 用 `runAfter` 投递一个延时回调，回调里要安全引用自己，需 `shared_from_this()` 保活。
- **为何 `channel_` 是 `unique_ptr` 且每次重连 `reset`**：每次 connect 都是新的 fd，旧 Channel 必须从 Poller 摘除并销毁，再为新 fd 建新 Channel。
- 连接成功后要**把 channel_ 摘掉**（连接的后续读写交给 TcpConnection 自己的 channel），再 `newConnectionCallback_(sockfd)`。

### 2.3 组件 1c：`TcpClient`

#### 职责
面向用户的客户端门面，对称于 `TcpServer`：内部持有一个 `Connector`，连接成功后构造 `TcpConnection`（**复用你已有的 TcpConnection**），并支持断线自动重连。

#### API
```cpp
// include/net/tcpclient.hpp
class TcpClient : noncopyable {
 public:
  TcpClient(EventLoop* loop, const InetAddress& serverAddr, std::string name);
  ~TcpClient();

  void connect();      ///< 发起连接（启动 connector_）
  void disconnect();   ///< 优雅关闭当前连接（shutdown 写端）
  void stop();         ///< 停止 connector_，不再重连

  TcpConnectionPtr connection() const;   ///< 当前连接（可能为空）
  EventLoop* getLoop() const { return loop_; }
  bool retry() const { return retry_; }
  void enableRetry()  { retry_ = true; }  ///< 断开后自动重连
  const std::string& name() const { return name_; }

  void setConnectionCallback(ConnectionCallback cb)       { connectionCallback_    = std::move(cb); }
  void setMessageCallback(MessageCallback cb)             { messageCallback_       = std::move(cb); }
  void setWriteCompleteCallback(WriteCompleteCallback cb) { writeCompleteCallback_ = std::move(cb); }

 private:
  void newConnection(int sockfd);                 ///< Connector 成功回调：建 TcpConnection
  void removeConnection(const TcpConnectionPtr&);  ///< 连接关闭：可触发重连

  EventLoop* loop_;
  ConnectorPtr connector_;
  const std::string name_;

  ConnectionCallback    connectionCallback_;
  MessageCallback       messageCallback_;
  WriteCompleteCallback writeCompleteCallback_;

  std::atomic<bool> retry_;
  std::atomic<bool> connect_;
  int nextConnId_;

  mutable std::mutex mutex_;        ///< 保护 connection_
  TcpConnectionPtr connection_;     ///< 当前连接，可能被多线程访问
};
```

#### 实现要点 / 易错点
- **`newConnection` 几乎照抄 `TcpServer::newConnection`**：建 `TcpConnection`、设四个回调（含内部 `closeCallback_` 指向 `removeConnection`）、调用 `connectEstablished()`。将来可把这段公共逻辑提炼，现在先复制无妨。
- **`connection_` 必须加锁**：客户端的 `connection()` 可能被用户从别的线程调用，而 `newConnection/removeConnection` 在 loop 线程改它。
- ⚠️ **析构竞态（重点）**：`TcpClient` 析构时，`connection_` 可能仍存活且运行在另一个 IO loop 上。推荐的处理方式：析构里若仍有连接，给它换一个特殊的 closeCallback（只做 `connectDestroyed`，不回调 TcpClient），再 `shutdown`。把这个时序想清楚再动手。
- **断线重连**：`removeConnection` 里若 `connect_ && retry_`，则 `connector_->restart()`。

#### ✅ 阶段一里程碑验收
- 写一个 `TcpClient` 连本机 echo-server：能连上、收发、`disconnect` 优雅关闭；杀掉服务端后客户端能按退避节奏自动重连。

## 3. 阶段二 · 传输扩展

### 3.1 组件 2a：`UdpServer`（数据报）

#### 职责
平行于 TcpServer，复用 `Channel/EventLoop/线程池`，但**不复用** `TcpConnection / Acceptor / Buffer`（理由见设计哲学：UDP 无连接、数据报、无 accept、无需累积）。

#### 回调与 API
```cpp
class UdpServer;
using UdpServerPtr = std::shared_ptr<UdpServer>;

/** @brief 收到一条 UDP 报文时触发，必须带源地址（无连接，来源各异）。 */
using UdpMessageCallback = std::function<void(
    UdpServer* server, const char* data, ssize_t len,
    const InetAddress& peer, Timestamp receiveTime)>;

class UdpServer : noncopyable {
 public:
  UdpServer(EventLoop* loop, const InetAddress& listenAddr, std::string name);

  void start();
  void setMessageCallback(UdpMessageCallback cb) { messageCallback_ = std::move(cb); }

  /** @brief 向指定对端发送一条报文（sendto）。 */
  void sendTo(const void* data, size_t len, const InetAddress& peer);

 private:
  void handleRead(Timestamp receiveTime);   ///< recvfrom 一条报文 → 回调

  EventLoop* loop_;
  std::string name_;
  std::unique_ptr<Socket>  socket_;          ///< SOCK_DGRAM
  std::unique_ptr<Channel> channel_;
  UdpMessageCallback messageCallback_;
  char recvBuf_[64 * 1024];                  ///< 单条报文上限（UDP 理论 64KB）
};
```

#### 实现要点
- **`Socket` 需支持 SOCK_DGRAM**：现有 `Socket` 默认 TCP。给 socket 创建逻辑加类型参数，或新增一个 `createNonblockingUdp()` 工厂。
- 一次 `recvfrom` 收一条完整报文进 `recvBuf_`，直接回调，**不累积、不用 Buffer**。
- 发送一般不阻塞（内核满了直接丢），先不做输出缓冲；要"可靠/限速"是上层协议的事（KCP/QUIC 思路），不在本组件。
- 可选进阶：connected UDP（对固定对端 `connect()` 后可用 `send/recv` 并收 ICMP 错误）。

### 3.2 组件 2b：Unix domain socket

#### 职责
本机进程间通信。**改动量极小**：传输路径（accept/读写/连接生命周期）完全复用 TCP 那套，只换地址族 `AF_UNIX` + `sockaddr_un` + 用文件路径代替 IP:port。

#### 最小改动策略
- 现在 `InetAddress` 只封装 `sockaddr_in`。两条路线选一：
  1. **轻量新增** `UnixAddress`（封装 `sockaddr_un`），让 `Acceptor/Connector/Socket` 的关键方法接受更抽象的地址（代价：要抽象地址接口）。
  2. **务实做法**：给 `Acceptor`/`Connector` 增加"以 unix 路径构造"的重载，bind/connect 时用 `sockaddr_un`，其余路径不变。
- 注意点：bind 前若路径文件已存在要 `unlink`；关闭时清理 socket 文件；`sun_path` 长度上限（约 108 字节）。

> 建议这一项排在 UDP 之后做，作为"复用现有路径"的练习，能加深你对"地址族是唯一变量、传输逻辑可复用"的理解。

### 3.3 组件 2c：`send` 重载 + zero-copy `sendFile`

#### TcpConnection 新增 API
```cpp
// include/net/tcpconnection.hpp 内补充
void send(const void* data, size_t len);   ///< 发送二进制数据，避免强制转 string
void send(Buffer* buf);                     ///< 直接吞掉一个 Buffer（swap/retrieve，零拷贝）

/**
 * @brief 用 sendfile(2) 把文件内容零拷贝发送（数据不经用户态）。
 * @param fileFd 已打开的文件 fd。
 * @param offset 起始偏移。
 * @param count  要发送的字节数。
 */
void sendFile(int fileFd, off_t offset, size_t count);
```

#### 实现要点
- 把现有 `sendInLoop(const std::string&)` 抽出底层 `void sendInLoop(const void* data, size_t len)`，三个 send 重载共用它。`send(Buffer*)` 用 `retrieveAllAsString` 或更优的 swap 避免拷贝。
- **`sendFile` 的背压（难点）**：`sendfile` 可能只发了一部分或返回 `EAGAIN`。完整版要在 TcpConnection 里记录"当前文件发送状态"（`fileFd / offset / remaining`），注册 `EPOLLOUT`，在 `handleWrite` 里继续 `sendfile` 直到发完。
  - **建议分两步**：先做"小文件一次发完"的简化版跑通；再加"大文件 + EPOLLOUT 续传"的完整背压版。
- 回顾：阶段一之前聊过的"基于 `writeCompleteCallback` 的分块文件下载"是**应用层**做法，不动底层就能流式传；本组件的 `sendFile` 是**下沉到底层的零拷贝优化**，两者可并存。

## 4. 阶段三 · 上层与运维

### 4.1 组件 3a：`LengthHeaderCodec`（分帧）+ Buffer 整数读写

#### 职责
**net 库之上的独立一层**，解决 TCP 字节流的粘包/半包：每条消息前加 4 字节长度头。**不进 TcpConnection**——它和"消息边界"这一应用协议约定绑定。

#### 需要给 Buffer 下沉的增强（唯一进底层的部分）
分帧需要按网络字节序读写整数与前置写入，这是协议无关的通用能力，可以进 Buffer：

```cpp
// include/base/buffer.hpp 内补充
int32_t peekInt32() const;       ///< 读 4 字节并 ntohl，不移动 readerIndex
int32_t readInt32();             ///< peekInt32 后 retrieve(4)
void    appendInt32(int32_t x);  ///< htonl 后 append
void    prepend(const void* data, size_t len);  ///< 写入 readerIndex 之前的 prependable 区
```
> 你现有 Buffer 已有 `prependableBytes()` 但没暴露写入口；`prepend` 正是利用这块预留空间，让"先写完消息体再在前面填长度"零拷贝完成。

#### Codec API
```cpp
class LengthHeaderCodec : noncopyable {
 public:
  /** @brief 解析出一条完整消息后回调给上层。 */
  using StringMessageCallback =
      std::function<void(const TcpConnectionPtr&, const std::string&, Timestamp)>;

  explicit LengthHeaderCodec(StringMessageCallback cb) : messageCallback_(std::move(cb)) {}

  /** @brief 注册给 conn->setMessageCallback；内部按长度头切分。 */
  void onMessage(const TcpConnectionPtr& conn, Buffer* buf, Timestamp receiveTime);

  /** @brief 发送：自动在前面加 4 字节长度头。 */
  void send(const TcpConnectionPtr& conn, const std::string& message);

 private:
  StringMessageCallback messageCallback_;
  static constexpr size_t kHeaderLen = sizeof(int32_t);
};
```

#### 实现要点
- `onMessage` 用 **`while` 循环**反复尝试切出完整消息：
  1. `readableBytes() < kHeaderLen` → `break`，等更多数据；
  2. 读出长度 `len = buf->peekInt32()`（先 peek 不消费）；做合法性检查（如 `len > 64MB` 视为错误，关连接）；
  3. `readableBytes() < kHeaderLen + len` → `break`，半包等数据；
  4. 否则 `retrieve(kHeaderLen)` 跳过头、`retrieveAsString(len)` 取一条消息、回调，继续循环（一次到达多条）。
- `send` 用 `appendInt32` 或先 append body 再 `prepend` 长度头。
- 字节序统一用 `htonl/ntohl`。

### 4.2 组件 3b：`signalfd` 优雅退出

#### 职责
把 `SIGINT/SIGTERM` 变成 fd 可读事件（和 `eventfd`、`timerfd` 同一套路），回调里 `loop->quit()`，实现 Ctrl-C 时不暴力终止、走完析构、关闭连接。

#### 思路（API 可轻量）
- `sigprocmask` 阻塞目标信号 → `signalfd(-1, &mask, SFD_NONBLOCK|SFD_CLOEXEC)` 拿 fd → 包成 `Channel` 注册读事件 → `handleRead` 里 `read` 出 `signalfd_siginfo`，根据信号类型执行退出回调。
- 可做成一个小封装 `SignalHandler`，或直接在 main loop 里挂一个 Channel。

### 4.3 组件 3c：心跳 / 空闲连接踢除

#### 职责
长连接服务必备：踢掉长时间无数据的连接，防资源泄漏。依赖阶段一的定时器。

#### 两档实现
- **入门版**：每条 TcpConnection 记 `lastActiveTime_`，`runEvery(1.0, ...)` 定时扫描所有连接，超时的 `shutdown`。简单但 O(n) 扫描。
- **进阶版（TimingWheel 时间轮）**：环形 `bucket` 数组，每个 bucket 是一个 `set<shared_ptr<Entry>>`；连接每次活跃就把自己的 Entry 重新塞进当前 bucket。每秒推进一格、丢弃最老 bucket——`shared_ptr` 引用计数归零时自动触发踢除，**O(1) 且无需遍历**。这是一个很精彩的设计，值得作为本库运维能力的压轴实现。
  - 数据结构示意：`Entry` 持有 `weak_ptr<TcpConnection>`，析构时若 `lock()` 成功则 `shutdown`；`bucket` 持有 `shared_ptr<Entry>`；连接的 context 里存 `weak_ptr<Entry>`。

> 顺带：这一项会暴露一个需求——给 `TcpConnection` 加 **context（`std::any`）** 以挂载 per-connection 状态（如 Entry 的 weak_ptr）。这本身也是通用库的常见能力，可在此一并补上：
> ```cpp
> void setContext(const std::any& ctx) { context_ = ctx; }
> std::any* getMutableContext() { return &context_; }
> const std::any& getContext() const { return context_; }
> ```
